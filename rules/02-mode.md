# 模式与音频规则

> 视频生成模式定义、音频字段写法、字幕共生强制规则。

---

## R1 两种模式

| 模式 | 用户提供 | 字幕 | AI负责 |
|------|---------|------|--------|
| 创作模式 | 主题/目标/时长/风格等文本 | ❌ 不生成 | 自动生成画面、组件、占位素材；配BGM |
| 口播模式 | 音频(.mp3/.wav/.m4a) + SRT字幕 | ✅ 必须有 | 严格按音频/SRT排版 |

**默认行为**：
- 创作模式：不允许 `subtitles`；`audio` 默认配BGM
- 口播模式：必须同时有 `audio` 和 `subtitles`

---

## R2 口播模式输入校验

| 项 | 规则 |
|----|------|
| 必须提供 | `.mp3/.wav/.m4a` 音频 + `.srt` 字幕 |
| 路径不存在/扩展名不支持 | 阻塞进入第二阶段，要求重新提供 |
| 路径含 `..` 等穿越字符 | 拒绝拷贝 |

**素材白名单（硬规则）**：
- ✅ 只允许：音频文件、字幕文件
- ❌ 严禁将 `.txt/.doc/.png/.jpg/.pdf/.mp4` 等作为创作依据
- ❌ 严禁擅自挑选素材（多个音频/SRT时必须跟用户确认）
- 例外：用户明确说"也用这张图片/这个文档"

---

## R3 音频字段写法

系统不区分录音/BGM——`audio` 字段就是"一个音频文件"。通过设置/不设置 `loop/fadeIn/fadeOut` 表达用法意图。

```jsonc
// 写法1：字符串简写（配音用法，向后兼容）
"audio": "./assets/voice.mp3"

// 写法2：对象形式（BGM用法，推荐）
"audio": {
  "path": "./assets/bgm.mp3",
  "loop": true,
  "fadeIn": 1,
  "fadeOut": 2
}
```

---

## R4 用法判定与字幕共生（强制）

### 判定矩阵

| 场景 | audio写法 | subtitles | 行为 |
|------|----------|-----------|------|
| 配音用法 | 字符串或对象只有path | ✅ 必须 | 不设loop/fadeIn/fadeOut |
| BGM用法 | 对象，含loop=true | ❌ 严禁 | 必须设loop/fadeIn/fadeOut |
| 静音 | 不写audio | ❌ 严禁 | — |

### 共生约束

| audio用法 | subtitles | 是否允许 |
|-----------|-----------|---------|
| 配音用法 | ✅ 有 | ✅ 允许 |
| BGM用法 | ❌ 无 | ✅ 允许 |
| 无audio | ❌ 无 | ✅ 允许 |
| 配音用法 | ❌ 无 | ❌ 不允许（云端拦截） |
| BGM用法 | ✅ 有 | ❌ 严禁 |
| 无audio | ✅ 有 | ❌ 绝对禁止 |

### 字幕规则（仅配音用法）

| 项 | 规则 |
|----|------|
| 创作/BGM模式生成subtitles | ❌ 严禁 |
| 字幕覆盖音频 | 最早start≈0，最晚end≈音频时长（容差±0.5s） |
| 字幕空白 | 字幕之间不允许>3s空白 |
| 字幕文本来源 | 必须100%来自用户SRT，严禁LLM改写/压缩/扩展 |
| 时间戳 | 严禁调整，必须和SRT原文一致 |
| 条目对应 | SRT有几条，subtitles就有几条，严禁合并或拆分 |
| 单条长度 | ≤25字（中文）/15词（英文） |
| emoji | 不允许 |

**转换格式**：
```jsonc
"subtitles": [
  { "start": 0.000, "end": 5.200, "text": "第一句字幕原文" }
]
```

## R5 创作模式默认配BGM（强制）

**规则**：创作模式启动时，LLM默认必须给视频配BGM——除非用户主动说"不要BGM"。

**自动选源策略**：
1. 不询问用户偏好风格
2. LLM根据视频主题/风格自动从 `templates/bgm/` 内置库选一首最匹配的
3. 直接写路径到project.json，设上 `loop:true, fadeIn:1, fadeOut:2`
4. 视频生成完成时简短告知："我给视频配了一段{风格}风的BGM，如果想换可以告诉我"

**BGM 风格匹配**：

| 视频主题/关键词 | 选用 BGM |
|---|---|
| AI / 编程 / 算法 / 数据 / 科技 | `tech-pulse` |
| 咖啡 / 餐饮 / 生活方式 / 温馨 | `warm-cafe` |
| 创业 / 成长 / 励志 / 教育 | `uplifting` |
| 企业 / B2B / 演示 / 商务 | `corporate` |
| 教程 / 科普 / Vlog / 轻松 | `light-pop` |
| 旅行 / 纪录片 / 大场面 | `cinematic` |
| 不明确 | `corporate`（默认） |

**BGM 路径写法**：
```jsonc
{
  "audio": {
    "path": "./assets/placeholders/bgm/{风格}.mp3",
    "loop": true,
    "fadeIn": 1,
    "fadeOut": 2
  }
}
```

**用户主动拒绝BGM**：不写audio字段，视频静音播出。

**BGM 不存在时**：静音播出，不强行写不存在的路径。

## R6 口播模式一切以SRT为准

**三大铁律**：

| 铁律 | 规则 | 违反后果 |
|------|------|---------|
| ① 字幕文本100%不动 | 严禁LLM改写、压缩、扩展、润色 | 口播说A，画面配B，严重失真 |
| ② 总时长以音频/SRT为准 | 视频总时长=音频时长=SRT最晚end，严禁自行调整 | 视频比音频短→内容截断；比音频长→末尾空播 |
| ③ 组件出现/消失匹配字幕意思 | 每个区域对应一段连续字幕，组件内容必须与当前字幕语义一致 | 画面讲A时字幕讲B，严重错位 |

**设计实践**：
1. 先读SRT，再划分区域（按SRT段落结构）
2. 组件内容紧扣当前字幕
3. 节奏门槛冲突时优先保SRT（接受warning）

---

## 严禁清单

- ❌ 让用户回答"你想用什么模式"这种开放式问题
- ✅ 确认式询问：LLM自动推断模式，用AskUserQuestion一次性确认
- ❌ 配音模式下漏掉subtitles字段
- ❌ BGM模式下生成subtitles数组
- ❌ 创作模式下LLM主动编造字幕内容
- ❌ 创作模式不配任何音频也不告知用户（除非用户明确拒绝BGM）
