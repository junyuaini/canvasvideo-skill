# 步骤4：区域设计

> 前置步骤：[步骤3：生成骨架JSON](03-skeleton-build.md)
> 下一步：[步骤5：生成区域JSON](05-region-build.md)

---

## 目标

为单个区域设计组件、布局、样式和时间轴。

---

## 输入

| 来源 | 说明 |
|------|------|
| 骨架配置 | `skeleton.json` 中该区域的信息 |
| 上一步产出 | 前序区域的 `design-P{n}.md`（如有） |
| 引用规则 | `rules/03-layout.md`、`rules/06-components.md`、`rules/05-richness.md`、`rules/04-timing.md` |

---

## 操作

### 第 1 步：读取模板

读取 `templates/artifacts/design-region.md`，了解输出格式。

### 第 2 步：确定区域内容

根据骨架设计中的区域主题，编写文案：

| 元素 | 内容 | 字数限制 |
|------|------|----------|
| 主标题 | 区域核心信息 | ≤ 15 字 |
| 副标题 | 补充说明 | ≤ 30 字 |
| 正文 | 详细内容 | ≤ 100 字 |
| 标签 | 关键词 | ≤ 8 字 |

**信息层级**：

```
层级 1（最重要）：主标题 — 最大字号、最醒目颜色
层级 2（重要）：副标题/核心数据 — 中等字号
层级 3（辅助）：标签、说明文字 — 小字号
层级 4（装饰）：图标、背景元素 — 不干扰阅读
```

### 第 3 步：设计布局

用 ASCII 绘制布局：

```
┌─────────────────────────────┐
│                             │
│     【主标题】               │  ← TitleComponent
│                             │
│  ┌─────────────────────┐   │
│  │    【核心内容】      │   │  ← ShockComponent / CardComponent
│  └─────────────────────┘   │
│                             │
│       [标签]                │  ← BadgeComponent
│                             │
└─────────────────────────────┘
```

**三种布局模式**（控制布局和视频显示逻辑）：
- **模式一（纯动态）**：所有组件都是普通组件，按时间依次出现，自动聚拢到视口中央，只需配置 w/h
- **模式二（纯固定）**：整个区域就是一个 AggregateComponent，内部所有子组件位置固定，需手动配置 x/y/w/h
- **模式三（混合）**：外层组件（普通组件 + AggregateComponent）按动态模式自动聚拢，内层 AggregateComponent 按固定模式手动配置位置

**硬规则**：单一模式占比不得超过 60%。如果总共 5 个区域，同一种模式最多用 3 个。

### 第 4 步：选择组件

查阅 `rules/06-components.md` 选型决策树。

| 组件类型 | 适用场景 | 数量限制 |
|----------|----------|----------|
| TitleComponent | 主标题 | 每区域 1 个 |
| TextComponent | 正文、说明 | 每区域 ≤ 2 个 |
| ShockComponent | 强调数据、金句 | 每区域 ≤ 1 个 |
| BadgeComponent | 标签、分类 | 每区域 ≤ 2 个 |
| ImageComponent | 图片展示 | 每区域 ≤ 1 个 |
| CardComponent | 结构化信息 | 每区域 ≤ 2 个 |
| GraphicComponent | 图表、流程 | 每区域 ≤ 1 个 |
| QuoteComponent | 名言、证言 | 每区域 ≤ 1 个 |

**原则**：单个区域组件总数 ≤ 5 个

**占位图必须用 Aggregate 包装**：当某个组件是 `[AI 自动生成 - 占位]` 的 ImageComponent 时，**必须**写成 AggregateComponent，children 包含 ImageComponent + ShockComponent（水印）。

> ⚠️ **注意**：AggregateComponent 的子组件**必须**包含完整字段（id、type、position、content、customStyle），详见 `rules/06-components.md` §R6。

### 第 5 步：设计时间轴

**使用全局绝对时间**（基于骨架设计中的区域时间范围）：

假设 P1 的时间范围是 0-10s：

| 组件 | start | end | 说明 |
|------|-------|-----|------|
| P1-001 | 0 | 10 | 全程显示 |
| P1-002 | 2 | 10 | 延迟出现 |

**原则**：
- 组件时间使用**全局绝对时间**（不是相对时间）
- 组件 start ≥ 区域 start
- 组件 end ≤ 区域 end
- 末组件停留：创作模式 ≤ 1s，口播模式 ≤ 2s
- 相邻组件间隔：创作模式 ≤ 1s，口播模式 ≤ 3s

**时间设计原则**：
- 组件入场有间隔（避免同时出现）
- 关键信息停留时间 ≥ 2 秒
- 字幕与对应组件同步

### 第 6 步：视觉丰富度检查

检查区域是否满足 `rules/05-richness.md` 门槛：

| 门槛 | 要求 |
|------|------|
| 门槛 1 | 至少 1 个彩色元素 |
| 门槛 2 | 至少 2 种字号 |
| 门槛 3 | 至少 1 个装饰元素（阴影、圆角、渐变） |
| 门槛 4 | 图片/图标占位符已标注 |

### 第 7 步：生成设计文档

```markdown
# 区域设计：P1 — {主题}

## 内容规划
- 主标题："..."
- 副标题："..."

## 布局设计
```
[ASCII 图]
```

## 组件清单
| ID | 类型 | 内容 | 位置 |

## 时间轴
| 组件 | start | end |
```

### 第 8 步：保存文件

```js
fs.writeFileSync(
  path.join(workdirRoot, skillProjectId, `design-${regionName}.md`),
  designContent
);
```

## 产出

| 文件 | 路径 | 说明 |
|------|------|------|
| design-P{n}.md | `{workdir}/{skillProjectId}/design-P{n}.md` | 区域设计文档 |

## 自检

- [ ] 组件总数 ≤ 5 个
- [ ] 每个组件有明确的 ID（格式：`{区域}-###`，如 P1-001）
- [ ] 位置坐标在 viewport 范围内
- [ ] 时间轴无重叠冲突
- [ ] 末组件停留符合模式要求
- [ ] 相邻组件间隔符合模式要求
- [ ] 视觉丰富度 4 门槛通过
- [ ] 占位图已用 Aggregate 包装（如有）
